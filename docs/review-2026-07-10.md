# NumericsSharp 项目审查报告

**日期：** 2026-07-10
**审查范围：** 全项目代码审查，评估对常规有限元分析线性方程求解的适配度

---

## 一、项目概览

| 维度 | 现状 |
|------|------|
| 项目结构 | 4 个托管库 + 1 个 C++ 原生层 + Benchmark 项目 |
| 目标框架 | .NET 8.0 |
| 原生依赖 | Intel MKL (PARDISO)，仅支持 win-x64 |

**分层架构：**

- **NumericsSharp.Core** — 稀疏矩阵 (CSR)、向量、组装器、Dirichlet 边界条件、残差计算
- **NumericsSharp.Solvers** — 迭代求解器 (CG/PCG) 与预条件器 (Jacobi/ILU0/IC0)
- **NumericsSharp.Mkl** — PARDISO 直接求解器封装 (含原生 C++ 适配层)
- **NumericsSharp.Mkl.Native** — C++ 桥接层，封装 MKL `pardiso` 函数

---

## 二、逐项能力评估

### 2.1 稀疏矩阵存储 — 良好，但格式单一

**已有：** CSR 格式 (`CsrMatrix`)，含行偏移/列索引/值三数组，实现了 `ILinearOperator` 接口的 `Multiply`。

**亮点：**
- `SparseMatrixBuilder` 支持 COO 式添加 + 自动合并重复条目，适合 FEM 单元组装
- `CsrMatrixPattern` + `SparseMatrixAssembler` 实现固定稀疏模式下的高效重复装配（非线性/瞬态 FEM 的关键需求）
- `SparseMatrixBuilder.AddSymmetricSubmatrix` 自动将上三角单元刚度矩阵展开为全矩阵

**不足：**
- 不支持 CSC 格式 — 某些预条件器（如 ILUT、SOR 后向替换）需要按列访问
- 不支持 dense matrix — 无法处理混合稀疏/稠密问题（如接触、约束增强）
- 不支持 block/vector-valued CSR — 固体力学中每个节点多自由度时，逐个标量组装效率低

### 2.2 直接求解器 — 仅 PARDISO，依赖原生库

**已有：** `PardisoSolver` 实现 `IDirectSparseSolver`，支持 Analyze → Factorize → Solve 三阶段分离。

**亮点：**
- 多 RHS 支持 (`rightHandSideCount` 参数)
- 矩阵类型自动识别（SPD 时仅取上三角，节省内存和计算）
- 错误上下文丰富：返回 phase、matrixType、order、nonZeroCount、pardisoErrorCode
- `NumericsThreadingOptions` 控制 MKL 内部线程数

**不足：**
- 仅 win-x64 可运行 — 原生 DLL 编译依赖 `MKLROOT` 环境变量，CMakeLists.txt 无 Linux/macOS 配置
- 无备用纯托管直接求解器 — 当 MKL 不可用时，无任何直接求解手段
- 无多前端/外存求解器 — 对于百万自由度以上规模无能为力

### 2.3 迭代求解器 — 仅 CG，严重不足

**已有：** CG (`ConjugateGradientSolver`)、PCG (`PreconditionedConjugateGradientSolver`)、门面类 `ConjugateGradientLinearSolver`

**不足（关键缺失）：**
- 无 GMRES — FEA 中最常用的非对称迭代求解器。流体力学 (Navier-Stokes)、对流-扩散问题、非关联塑性均产生非对称矩阵
- 无 MINRES — 对称不定矩阵（如混合有限元、鞍点问题）需要 MINRES
- 无 BiCGStab — 非对称矩阵的内存友好替代方案
- 无多重网格 (Multigrid) — 大规模问题 CG 收敛极慢，需要几何或代数多重网格预处理
- 无 Krylov 子空间基类 — 所有迭代求解器硬编码实现，缺乏对残差历史、重启、正交化策略的抽象

### 2.4 预条件器 — 基础覆盖，性能不足

**已有：** Jacobi、ILU0、IC0

**亮点：**
- `IPreconditioner` 接口简洁 (`Order` + `Apply(residual, result)`)
- ILU0/IC0 的因式分解逻辑正确

**不足：**
- Jagged array 存储 (ILU0/IC0) — `int[][]`/`double[][]` 在 10 万阶以上矩阵中内存碎片严重，应使用扁平 CSR 存储
- 无 ILUT/ILUTP — ILU0 对许多实际问题精度不够（drop tolerance 控制是关键）
- 无 AMG (代数多重网格) — 大规模椭圆型问题（结构静力、热传导）的标配
- 无分块/区域分解预条件器 — 并行计算的基础
- 无 SOR/SSOR — 最简单有效的对称预条件器

### 2.5 边界条件 — 仅 Dirichlet

**已有：** `DirichletBoundaryCondition.Apply` — 正确实现了消元法（移除非约束行列 + 修改 RHS）

**不足：**
- 无 Neumann/Robin 边界条件 — 需要在组装阶段处理
- 无多点约束 (MPC/rigid links) — 许多工程问题需要
- 无周期边界条件
- `Apply` 方法总是创建新矩阵 — 在固定稀疏模式下可以就地修改以提升性能

### 2.6 组装与 FEM 集成 — 缺失

- 无单元库 — 没有杆、梁、三角形、四边形、四面体、六面体等单元的形函数和刚度矩阵计算
- 无材料模型 — 没有弹性、塑性、超弹性等本构
- 无节点/自由度映射 — 用户必须自行管理节点编号和自由度编号
- 无网格导入 — 不支持 Gmsh、Abaqus .inp 等常见格式
- 无载荷/约束的高层抽象 — 目前用户需手动构造 RHS 和 indices/values

### 2.7 并行计算 — 仅 MKL 内部并行

**已有：** `NumericsThreadingOptions` 区分 `NativeInnerParallel` 和 `ManagedOuterParallel` 模式

**不足：**
- `ManagedOuterParallel` 模式虽然定义了接口，但无任何托管代码使用它 — `ConjugateGradientSolver`、预条件器、矩阵组装全部是单线程
- 稀疏矩阵向量乘 (SpMV) 是迭代求解器的核心瓶颈，必须多线程才能利用多核 CPU
- 组装阶段（单元循环）是最容易并行的部分，目前完全串行

### 2.8 数值质量 — 良好

- 所有公共 API 均有完备的实参验证
- `SparseMatrixBuilder` 对重复条目求和（符合 FEM 组装语义）
- `DirichletBoundaryCondition` 正确处理了重复约束检测
- CG 和 PCG 实现了正确的 breakdown 检测（分母 ≤ 0 或非有限）
- PARDISO 求解后计算残差范数进行质量检验

## 四、总体评估

**当前状态：满足小型 SPD 问题的基本求解需求，但不满足"常规有限元分析"的完整要求。**

项目已经建立了良好的架构基础和正确的数值实现，具备以下可用路径：

- 小规模（<10 万自由度）弹性静力问题：`SparseMatrixBuilder` 组装 → `PardisoSolver` 求解 → 可工作
- SPD 瞬态/非线性问题：`SparseMatrixAssembler` 固定模式装配 → `PardisoSolver.Factorize` 复用符号分解 → 可工作

三个关键缺口导致"常规 FEA"定位不成立：

1. **无法求解非对称系统** — 流体、对流-扩散、非关联塑性全部排除
2. **无法在多平台上运行** — 仅 Windows + MKL
3. **大规模问题性能不足** — 单线程 SpMV + 低效预条件器

## 五、改进建议优先级

| 优先级 | 项目 | 说明 |
|--------|------|------|
| P0 | GMRES + BiCGStab 迭代求解器 | 扩充非对称矩阵求解能力 |
| P0 | ILUT 预条件器 + ILU0/IC0 扁平数组重构 | 提升预条件器精度和内存效率 |
| P1 | 多线程 SpMV | 迭代求解器性能瓶颈，利用多核 CPU |
| P1 | 跨平台原生层或纯托管直接求解器 | 解除 Windows 单平台限制 |
| P2 | MINRES 求解器 | 对称不定矩阵（混合有限元、鞍点问题） |
| P2 | AMG 预条件器 | 大规模椭圆型问题收敛加速 |
| P2 | 多线程单元组装 | 充分利用多核进行 FEM 组装 |
| P3 | Neumann/Robin 边界条件 | 完善边界条件体系 |
| P3 | 单元库与材料模型 | 降低 FEM 使用门槛 |
| P3 | 网格导入 (Gmsh/Abaqus) | 打通完整工作流 |
